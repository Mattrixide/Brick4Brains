This document captures all functional requirements derived from the Product Requirements Document. Each requirement is traceable back to a PRD section, assigned a priority level (P0 = MVP, P1 = Phase 2, P2 = Phase 3), and includes acceptance criteria. Click any row to expand details.

FR-CAL-001 Guided 4-corner calibration via ArUco markers P0 PRD 2.1
Description
System shall support a guided calibration flow where the operator places a single ArUco tag at each of the four arena corners sequentially. The system prompts for each corner placement in order and captures the position.
Acceptance Criteria
- System displays clear prompts for each corner (1 through 4) in sequence
- Each corner is captured when the ArUco marker is detected and confirmed
- All four corners must be captured before calibration is considered complete
- Visual feedback confirms each successful capture
Source
PRD Section 2.1 — Arena Calibration
FR-CAL-002 Compute homography transform from camera to arena coordinates P0 PRD 2.1
Description
After capturing all four corners, the system shall compute a homography transform that maps camera pixel coordinates to arena coordinates (0,0 to 8,8 in feet).
Acceptance Criteria
- Homography matrix is computed from the 4 corner correspondences
- Any pixel within the arena can be mapped to arena coordinates with <0.5 inch accuracy
- Transform is validated against known positions before use
Source
PRD Section 2.1 — Arena Calibration
FR-CAL-003 Identify pit location via dedicated ArUco marker P0 PRD 2.1
Description
System shall identify the pit location and boundary using a dedicated ArUco marker (ID 4) placed at or near the pit during calibration.
Acceptance Criteria
- Pit ArUco marker (ID 4) is detected and its position recorded
- Pit boundary is established in arena coordinates
- Pit zone is available to the strategy engine for avoidance logic
Source
PRD Section 2.1 — Arena Calibration; PRD Section 7.2 — ArUco Markers
FR-CAL-004 Persist calibration data within a session P0 PRD 2.1
Description
Calibration data (corner positions, homography matrix, pit location) shall persist so it survives restarts within a session.
Acceptance Criteria
- Calibration data is saved to disk after successful calibration
- On restart, previously saved calibration is loaded automatically
- User can verify loaded calibration is still valid
Source
PRD Section 2.1 — Arena Calibration
FR-CAL-005 Manual adjustment of calibration via web dashboard P0 PRD 2.1
Description
The web dashboard shall allow manual adjustment of arena bounds post-calibration and support re-calibration without full system restart.
Acceptance Criteria
- Corner positions can be adjusted via drag-and-drop on the arena map
- Pit boundary can be adjusted
- Re-calibration can be triggered from the dashboard
- Calibration profiles can be saved and loaded
Source

FR-CAL-006 Capture clean background reference frame during calibration P0 PRD 7.3
Description
During calibration (with no robots in the arena), the system shall capture a clean background reference frame for use in enemy detection via frame differencing.
Acceptance Criteria
- A clean frame is captured when no robots are present
- Background reference is stored for use by the enemy detection pipeline
- System prompts the user to clear the arena before capture

FR-TRK-001 Detect ArUco tag position from overhead camera P0 PRD 2.2
Description
System shall detect our robot's ArUco tag (ID 5, top surface) from the ceiling-mounted OAK-D Pro camera and determine position (x, y) in arena coordinates using the calibrated homography.
Acceptance Criteria
- ArUco ID 5 is reliably detected from 1080p feed at 7ft height
- Position is reported in arena coordinates (0-8ft range)
- Detection works with 3-inch markers appearing as ~60px
Source
PRD Section 2.2 — Robot Tracking; PRD Section 7.2 — ArUco Markers
FR-TRK-002 Determine robot heading from ArUco tag rotation P0 PRD 2.2
Description
System shall extract heading angle (orientation) from the detected ArUco tag's rotation in the camera image.
Acceptance Criteria
- Heading angle is extracted from ArUco marker corners
- Heading is reported in degrees or radians relative to arena coordinate system
- Accuracy sufficient for strategy engine decision-making
Source
PRD Section 2.2 — Robot Tracking
FR-TRK-003 Handle temporary ArUco occlusion with prediction P0 PRD 2.2
Description
When ArUco detection fails (e.g., robots colliding, dust), the system shall predict/extrapolate position using last known velocity for up to 500ms.
Acceptance Criteria
- Position is extrapolated using last known velocity during occlusion
- Prediction is maintained for up to 500ms
- After 500ms without detection, position is flagged as uncertain
- No erratic behavior from a single dropped frame
Source
PRD Section 2.2; PRD Section 7.4 — Occlusion Handling
FR-TRK-004 Bottom ArUco tag as backup for flipped robot P0 PRD 2.2
Description
If our robot flips and the top ArUco tag (ID 5) is not visible, the system shall detect the bottom ArUco tag (ID 6) as a backup.
Acceptance Criteria
- ArUco ID 6 is recognized as our robot's bottom marker
- Position and heading are derived from ID 6 when ID 5 is not detected
- System correctly infers that the robot is flipped
Source
PRD Section 2.2 — Robot Tracking; PRD Section 7.2 — Marker IDs
FR-TRK-005 Complementary filter for camera + IMU fusion (MVP) P0 PRD 7.4
Description
For the MVP, system shall use a complementary filter to fuse camera-based position/heading with IMU gyro data for higher-frequency orientation updates between camera frames.
Acceptance Criteria
- High-pass filter on IMU gyro for fast heading changes
- Low-pass filter on camera heading for absolute reference
- Effective pose update rate of at least 30 Hz
Source
PRD Section 2.2; PRD Section 7.4 — State Estimation
FR-TRK-006 EKF-based sensor fusion for camera + IMU P1 PRD 7.4
Description
Upgrade from complementary filter to Extended Kalman Filter for camera + IMU fusion. State vector: [x, y, vx, vy, heading, omega].
Acceptance Criteria
- EKF prediction step uses IMU data at 100-200 Hz
- EKF correction step uses camera measurements at 30-60 Hz
- State vector includes position, velocity, heading, and angular velocity
- Runs in <1ms per update
Source
PRD Section 7.4 — State Estimation (Phase 2)
FR-ENM-001 Detect enemy robot via depth thresholding P0 PRD 2.3
Description
System shall detect the enemy robot by thresholding the OAK-D Pro depth map: any object elevated more than 2cm above the floor plane is a candidate robot.
Acceptance Criteria
- Depth map from 400P active stereo mode is processed
- Floor plane depth is established during calibration
- Objects >2cm above floor are identified as candidates
- Active stereo IR projector helps with textureless white floor
Source
PRD Section 2.3; PRD Section 7.3 — Enemy Detection Strategy
FR-ENM-002 Detect enemy robot via reference frame differencing P0 PRD 7.3
Description
System shall detect the enemy by comparing live frames against the clean background reference captured during calibration (frame differencing).
Acceptance Criteria
- Live frames are differenced against calibration background reference
- Moving objects are identified from significant pixel differences
- Logo area is masked to prevent false positives
- Shadows are handled (via MOG2 shadow detection or depth verification)
Source
PRD Section 7.3 — Enemy Detection Strategy (MVP)
FR-ENM-003 Disambiguate enemy from our robot using known position P0 PRD 2.3
Description
When multiple objects are detected, the system shall exclude our robot's known position/footprint to isolate the enemy. The largest remaining contour is identified as the enemy.
Acceptance Criteria
- Our robot's known position is masked from candidate detections
- Exclusion zone is sized appropriately for the robot footprint
- Largest remaining contour is classified as the enemy
- Works when robots are adjacent or overlapping
Source
PRD Section 2.3 — Enemy Robot Tracking
FR-ENM-004 Track enemy position and estimate heading from motion vector P0 PRD 2.3
Description
System shall track the enemy position (x, y) in arena coordinates and estimate its heading from the velocity vector direction over recent frames.
Acceptance Criteria
- Enemy position is reported in arena coordinates
- Heading is estimated from the direction of motion
- Velocity and acceleration are estimated for strategic prediction
- Works with varying robot shapes and colors
Source
PRD Section 2.3 — Enemy Robot Tracking
FR-ENM-005 Kalman filter tracking for enemy position smoothing P1 PRD 7.3
Description
Add Kalman filter on enemy centroid position [x, y, vx, vy] to smooth noisy detections, predict through brief occlusions, and provide velocity estimates.
Acceptance Criteria
- Kalman filter state: [x, y, vx, vy]
- Smooths frame-to-frame jitter in detection
- Predicts through brief occlusions (<500ms)
- Provides velocity estimate to strategy engine
Source

FR-ENM-006 On-device neural network enemy detector P2 PRD 7.3
Description
Train a custom object detector (MobileNet-SSD or YOLOv5-nano) on combat robot images and deploy to the OAK-D Pro Myriad X VPU for on-device enemy detection at ~30 FPS.

- Model deployed to Myriad X VPU via OpenVINO
- Runs at 30+ FPS with minimal host CPU impact
- Detects various robot shapes and colors

FR-STR-001 Basic pursuit behavior: drive toward enemy P0 PRD 2.4
Description
The strategy engine shall implement a Pursue behavior that closes distance to the enemy robot by computing motor commands to drive toward the enemy's current position.
Acceptance Criteria
- Robot drives toward enemy's detected position
- Motor commands are appropriate differential drive values
- Heading correction applied to maintain approach vector
Source
PRD Section 2.4 — Strategy Engine (Pursue behavior)
FR-STR-002 Push behavior: drive enemy toward the pit P0 PRD 2.4
Description
The strategy engine shall implement a Push behavior that drives into the enemy to push it toward the pit corner, using the pit location from calibration.
Acceptance Criteria
- Push direction computed based on enemy position relative to pit
- Robot applies sustained force toward pit
- Behavior activates when in contact range with enemy
Source
PRD Section 2.4 — Strategy Engine (Push behavior)
FR-STR-003 Pit avoidance: never drive our robot into the pit P0 PRD 2.4
Description
The strategy engine shall always prevent our robot from driving into the pit. This is a hard constraint that overrides all other behaviors.
Acceptance Criteria
- Pit zone boundary is checked before any motor command
- If commanded path would enter the pit zone, override with safe trajectory
- No scenario causes our robot to self-eliminate
Source
PRD Section 2.4 — Strategy Engine (Avoid pit)
FR-STR-004 Strategy decisions at 10+ Hz P0 PRD 2.4
Description
The strategy engine shall produce motor command decisions at a rate of at least 10 Hz to maintain responsive behavior.
Acceptance Criteria
- Strategy loop runs at 10 Hz or faster
- Motor commands are updated every 100ms or less
- No decision cycle takes more than 100ms

PRD Section 5 — Phase 3 Additions
FR-COM-001 WiFi UDP motor command transmission P0 PRD 2.5
Description
System shall transmit motor commands to the ESP32 over WiFi UDP using a binary struct packet format (8 bytes: cmd_type, sequence, left_speed, right_speed, checksum).
Acceptance Criteria
- Motor commands sent via UDP to ESP32
- Binary packet format with checksum validation
- Differential drive commands: left_speed and right_speed
- Commands sent at 50 Hz
Source
PRD Section 2.5; PRD Section 7.5 — Communication Protocol
FR-COM-002 Motor command latency under 20ms RTT P0 PRD 2.5
Description
Round-trip latency for motor commands shall be under 20ms under normal conditions.
Acceptance Criteria
- Median RTT <20ms measured under arena conditions
- WiFi power saving disabled on ESP32
- Dedicated WiFi AP used (not shared)
- Nagle's algorithm disabled
Source
PRD Section 2.5; PRD Section 7.5 — Communication Protocol
FR-COM-003 Emergency stop command P0 PRD 2.5
Description
System shall support an emergency stop command that immediately halts all motors on the ESP32.
Acceptance Criteria
- Emergency stop command type defined in protocol
- ESP32 immediately sets all motor speeds to zero on receipt
- Emergency stop can be triggered from web dashboard
Source
PRD Section 2.5 — ESP32 Communication
FR-COM-004 200ms communication failsafe timeout P0 PRD 2.5
Description
ESP32 firmware shall stop all motors if no valid motor command is received within 200ms (watchdog failsafe).
Acceptance Criteria
- ESP32 implements 200ms watchdog timer
- Both motors set to zero speed on timeout
- Timer resets on each valid command received
- Failsafe activates within 200ms of last valid command
Source
PRD Section 2.5; PRD Section 7.5 — Failsafe
FR-COM-005 IMU telemetry stream from ESP32 at 100+ Hz P0 PRD 2.5
Description
ESP32 shall stream IMU telemetry (gyro, accelerometer) to the PC over WiFi UDP at 100 Hz or higher, along with battery voltage and heartbeat.
Acceptance Criteria
- IMU data transmitted at 100+ Hz on separate UDP port
- Packet includes gyro (3-axis), accelerometer (3-axis), heading, battery voltage
- Binary struct format with sequence number and checksum
- Connection health monitored via heartbeat
Source

Acceptance Criteria

Failsafe behaviors (motor stop) shall activate within 200ms of any critical failure: camera frame drops, WiFi interruptions, or ArUco detection failures.
Acceptance Criteria
